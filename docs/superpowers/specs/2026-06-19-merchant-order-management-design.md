# Merchant Order Management + Simulated Order Data — Design

**Date:** 2026-06-19
**Status:** Approved for implementation
**Scope:** Backend (`sky-server`) only. Real WeChat Pay is explicitly out of scope.

## 1. Goal

Make the merchant (admin) side able to manage the full order lifecycle, and seed the
database with realistic orders across all statuses so the feature is demonstrable.

Two parts:
- **Part A** — Simulated order seed data (SQL).
- **Part B** — Merchant order management API.

Payment is **not** implemented. Simulated orders are written directly in already-paid
states (`pay_status = 1`), so the merchant flow (accept → deliver → complete, or reject/cancel)
can run end-to-end without a payment step.

## 2. Background / Current State

- `OrderService` declares only `submitOrder`; `OrderServiceImpl` implements only that.
- `OrderMapper` has only `insert`. (`OrderMapper.xml` exists for the insert.)
- No `admin/OrderController`.
- DTOs/VOs already exist and are reused as-is:
  - `OrdersPageQueryDTO` (page, pageSize, number, phone, status, beginTime, endTime, userId)
  - `OrdersConfirmDTO` (id, status)
  - `OrdersRejectionDTO` (id, rejectionReason)
  - `OrdersCancelDTO` (id, cancelReason)
  - `OrderStatisticsVO` (toBeConfirmed, confirmed, deliveryInProgress)
  - `OrderVO extends Orders` (orderDishes, orderDetailList)
- `Orders` status constants: `PENDING_PAYMENT=1`, `TO_BE_CONFIRMED=2`, `CONFIRMED=3`,
  `DELIVERY_IN_PROGRESS=4`, `COMPLETED=5`, `CANCELLED=6`. Pay status: `UN_PAID=0`, `PAID=1`, `REFUND=2`.
- Seed-able FKs in DB: `user.id = 4`, `address_book.id = 2` (user_id=4, phone 13712341234, detail "12345").
- Dishes available from the earlier international-dish seed (TheMealDB names/images/prices).

## 3. Part A — Simulated Order Data

A re-runnable, dedupe-guarded SQL file: `db/seed-orders.sql`.

**Volume & status spread** (about 12 orders so every management action has a target):
- `TO_BE_CONFIRMED (2)` — 4 orders (paid, awaiting merchant accept)
- `CONFIRMED (3)` — 2 orders (accepted, awaiting delivery)
- `DELIVERY_IN_PROGRESS (4)` — 2 orders
- `COMPLETED (5)` — 3 orders (spread over recent days, for realistic history)
- `CANCELLED (6)` — 1 order (with cancel_reason)

**Each order:**
- `user_id = 4`, `address_book_id = 2`.
- `number` = unique timestamp-like string.
- `pay_method = 1`, `pay_status = 1` (PAID) for all non-cancelled; cancelled also paid then refunded is out of scope — cancelled order uses `pay_status = 1` with a `cancel_reason`.
- `user_name`, `phone`, `consignee`, `address` filled from the address book / sensible defaults.
- `amount` = sum of its order_detail line amounts (+ optional pack_amount).
- Timestamps (`order_time`, `checkout_time`, `delivery_time`, etc.) consistent with the status.
- 2–3 `order_detail` rows referencing real seeded dishes (name, image, dish_id, number, amount).

**Dedupe:** guarded by `orders.number` via `WHERE NOT EXISTS`. Generated by a Python script
(`/tmp/gen_orders.py` style) that reads available dish ids from the DB, mirroring the dish-seed approach.

## 4. Part B — Merchant Order Management API

New `admin/OrderController` (`@RequestMapping("/admin/order")`), backed by additions to
`OrderService` / `OrderServiceImpl` and new `OrderMapper` methods.

### Endpoints

| Method & path | Purpose | Input | Output |
|---|---|---|---|
| `GET /admin/order/conditionSearch` | Paged conditional search | `OrdersPageQueryDTO` | `Result<PageResult>` of `OrderVO` |
| `GET /admin/order/statistics` | Count of orders in actionable statuses | — | `Result<OrderStatisticsVO>` |
| `GET /admin/order/details/{id}` | Order detail incl. line items | path `id` | `Result<OrderVO>` |
| `PUT /admin/order/confirm` | Accept order (2 → 3) | `OrdersConfirmDTO` | `Result` |
| `PUT /admin/order/rejection` | Reject order (2 → 6) | `OrdersRejectionDTO` | `Result` |
| `PUT /admin/order/cancel` | Cancel order (→ 6) | `OrdersCancelDTO` | `Result` |
| `PUT /admin/order/delivery/{id}` | Start delivery (3 → 4) | path `id` | `Result` |
| `PUT /admin/order/complete/{id}` | Complete (4 → 5) | path `id` | `Result` |

### Service methods (added to `OrderService`)
- `PageResult conditionSearch(OrdersPageQueryDTO)` — uses PageHelper; for each order, populate
  `orderDishes` (a concatenated "name*number" summary) and `orderDetailList`.
- `OrderStatisticsVO statistics()` — counts of `TO_BE_CONFIRMED`, `CONFIRMED`, `DELIVERY_IN_PROGRESS`.
- `OrderVO details(Long id)` — order + its order_detail list.
- `void confirm(OrdersConfirmDTO)` — set status `CONFIRMED`.
- `void rejection(OrdersRejectionDTO)` — only allowed from `TO_BE_CONFIRMED`; set `CANCELLED` + reason + cancel_time.
- `void cancel(OrdersCancelDTO)` — set `CANCELLED` + reason + cancel_time.
- `void delivery(Long id)` — only from `CONFIRMED`; set `DELIVERY_IN_PROGRESS`.
- `void complete(Long id)` — only from `DELIVERY_IN_PROGRESS`; set `COMPLETED` + delivery_time.

### Mapper additions (`OrderMapper` + `OrderMapper.xml`)
- `Page<Orders> pageQuery(OrdersPageQueryDTO)` — dynamic where on number/phone/status/time range, order by order_time desc.
- `Orders getById(Long id)`.
- `void update(Orders orders)` — dynamic update of status / reason / times.
- `Integer countStatus(Integer status)`.
- `OrderDetail` lookups via existing `OrderDetailMapper.getByOrderId(Long)` (add if missing).

### Status transition rules & error handling
- Valid transitions: `2→3` (confirm), `2→6` (reject), `3→4` (delivery), `4→5` (complete), any→`6` (cancel, business-permitting).
- Operating on a non-existent order or from an illegal current status → throw `OrderBusinessException`
  with a clear message; controller returns an error `Result` (existing global exception handler).
- `delivery`/`complete` validate the current status before updating.

## 5. Testing (TDD)

Service-layer unit tests written before implementation, using the project's existing test setup
(Spring Boot test + a test DB or mocked mappers — follow whatever pattern exists; if none, use
Mockito on the mappers):
- `statistics()` returns correct counts for the seeded spread.
- `conditionSearch` filters by status and paginates; populates `orderDetailList`.
- `confirm` moves `2 → 3`.
- `rejection` from `2 → 6` sets reason; from a non-`2` status throws.
- `delivery` from `3 → 4`; from non-`3` throws.
- `complete` from `4 → 5`; from non-`4` throws.
- `cancel` sets `CANCELLED` + reason.
- `details` returns order with its line items.

Manual verification: run the seed, hit each endpoint via Knife4j (`/doc.html`) or curl, confirm
status transitions persist.

## 6. Out of Scope
- Real WeChat Pay / payment callbacks.
- User-side order endpoints (history, detail, cancel, repeat) — separate effort.
- Reports/statistics and Workspace modules — separate effort.
- WebSocket order reminders.
